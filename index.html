The Quad Tree

I implemented this data structure winter quarter of my freshman year in an effort to render sprites to a mobile device screen as efficiently as possible (after all, more efficiency = more FPS). While I never got around to making the game in Android I wanted to use this with, I still have this to be proud of.

The quad tree is a data structure that is very similar to the binary tree. It is made up of nodes, which have children and parents. However, each node in a quad tree has four children instead of two like a binary tree. The quad tree is commonly used to divide 2 dimension euclidean space. The root of the tree represents the entire space. Each child represent one fourth of its parent, dividing the parent across the middle both vertically and horizontally. This process continues with the children until a minimum size is reached. The quad tree proves its effectiveness in one particular area of game design: collision detection. Every sprite (or image) in a game that interacts with others sprites must be surrounded by a rectangle called a bounding box. This bounding box is checked each cycle with the other sprites bounding boxes to see if there are any overlaps a.k.a collisions. This can become very time consuming very quickly as the number of sprites increase. Since each sprite would have to check all other sprites, this procedure yields a worst case complexity of O( n * [n-1]) or O( n2 ).

The quad tree solves this problem by dividing up the coordinate space and assigning each sprite a parent node. The parent node is the smallest node in the tree that can fully contain the sprite's bounding box inside it. Sprites then only need to check for collisions with sprites with the same parent and sprites whose parents are the parent or ancestor of their parent node. In typical scenarios, where sprites are spread rather evenly across the map, this drastically reduces the number of collision checks made each cycle.

In the example shown in the photo, there are 2,000 sprites being rendered and moved across the screen. Rather than each sprite checking the other 1,999, they each check only those in their parent and its ancestors. Sprites that are not colliding are drawn black. The others are drawn in different colors at random. The nodes themselves are highlighted if they are not empty as blue boxes. In this particular case, the simulation could run at roughly 17 - 22 ms per cycle (~60 FPS), whereas the brute force technique could barely achieve 200 - 300 ms per cycle (under 5 FPS).

Steganography

Steganography is a process of embedding information (like a message) in a communication medium such that only people looking for the hidden information would be able to find it.

It differs from encryption in that encryption is easily noticeable (you can very easily tell when a file has been encrypted due to the scrambled contents of the file), but it is next to impossible to decipher without the encryption key. Conversely, the goal of steganography is not to make the information indecipherable, but unnoticeable. For this reason, digital steganography is typically done on images, where changing the red, green, or blue values by only a slight amount would produce a nearly identical image to the human eye, but would contain binary information that could be extracted by the intended recipient.

This program works by changing the least significant bit of all available color channels (RGB or ARGB) in a BMP or PNG image to encode a text message within the image. The user can enter an optional key for further security. The message can only be recovered using the extractor in the program and the right key. The image above has some steganographic text embedded in it. Try downloading the image, the software, and run it with the highly secure key 'password'.

To learn more about steganography, check out the Wikipedia Page on it.

Sky Pi

August 9th, 2014

So my family and I are heading to the beach for vacation this summer. Like always when going to the beach, my dad and I plan on doing a lot of kite flying. In the past, we attempted to bring a small video camera up with a kite and get some photos of the beach. As you can see from the photo we had mild success with that project. The video camera was cheap and we had no way of knowing what it was actually filming at any time. In fact, a majority of the footage we got was garbage, all we could do was extract individual frames from the video clips to get some neat still photos. This time, we want to take it a step further and see if we can improve the project.

Photo taken from a cheap video camera tied to a kite

The first step for this was planning. We knew we needed to be able to send information from the kite back down to the ground in order to stream photos, but we also wanted to have the option to pan or tilt the camera on the kite from the ground. We also knew we wanted to get the kite much higher (over 500ft) than last time. My dad took charge of purchasing and constructing the kite and harness for the camera, my job was implementing the hardware and software for the project to run. I settled on using Raspberry Pis, due to their cost, computing power, and huge amount of available libraries and support. One Raspberry Pi wasn't going to be enough, though. We needed one on the ground as a base station to communicate with the kite. We decided one using the XBee wireless modules for wireless communication between the two Pis. The XBee modules are based on the ZigBee protocol, a wireless network stack that allows for point-to-point and mesh wireless communication between microcontrollers. We settled on the XBee Pro 900, because it runs on the 900 MHz band instead of the crowded 2.4 GHz band. These wireless modules have a theoretical range of 1 mile (1500m) and can transmit about 250 Kbps, perfect for sending low resolution photos from the kite down to the base station.

Another aspect to consider was what to do with the data once it got to the base station. I decided the simplest solution was to create a wireless access point using the Pi. Basically, the base station Raspberry Pi would behave like a wireless router and create a network for other devices to connect to. Once connected to the Wi-Fi network, any device could view the photos downloaded from the kite using the base station pi. Commands could also be sent up to the kite to pan or tilt the camera using a web-interface hosted by the base station.
